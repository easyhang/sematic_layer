# 语义层构建助手

使用本 Skill 帮助开发者构建业务语义层：把业务语言映射到数据表，通过 L0 原子实体、最多两层概念层、SQL 模板和解释规则，让智能体能够稳定完成 `用户问题 → 概念 → 实体 → SQL → 数据事实 → 业务回答`。

## 适用场景

使用本 Skill，当用户需要：

- 从数据表、DDL、字段说明中抽象业务语义层。
- 判断当前信息是否足够形成可靠语义层。
- 将业务问题映射到概念、L0 原子实体和 SQL。
- 设计或审查 L0 原子实体、L1 业务概念、L2 决策概念。
- 生成语义层目录、YAML / JSON 定义、SQL 模板、Query 映射和测试用例。
- 在项目根目录生成一份实体信息汇总 HTML，集中展示所有 L0 原子实体、L1 概念和 L2 概念。
- 检查语义层是否存在口径臆造、概念层过深、SQL 不可执行、依赖循环或敏感字段风险。

## 核心工作流

1. 先做结构化思考，不要直接建模。
2. 判断用户提供的信息是否足够。
3. 如果 SQL 正确性依赖缺失信息，先追问；如果只是权限边界缺失，可继续生成候选语义层并标记风险。
4. 识别业务域、使用者、决策场景和典型 Query。
5. 解析数据表、字段、主外键、表关系、数据粒度、时间字段、状态过滤和指标口径。
6. 抽象可落到 SQL 的 L0 原子实体。
7. 构建最多两层概念：
   - L1 业务概念：由 L0 原子实体组合而成。
   - L2 决策概念：由 L1 概念组合而成，必要时可少量直接依赖 L0 原子实体。
8. 生成 `用户问题 → 概念 → 实体 → SQL` 映射样例。
9. 默认输出到 `semantic-layer/{domain}/`，除非项目已有更明确的目录约定。
10. 在项目根目录生成 `semantic-layer-entities.html`，汇总展示 `entities/atomic.yaml` 和 `entities/concepts.yaml` 中的实体信息。
11. 生成结构校验和行为测试用例。
12. 输出假设、风险、缺失信息和需要确认的问题。

## 结构化思考协议

在语义层设计、重大口径选择、概念边界划分、输出格式设计或架构取舍时，使用以下思考框架。不要把它当成冗长模板机械输出；把它作为内部推理和必要时的显式报告结构。

### 步骤 1：先收集上下文

在推理之前，先收集已存在的信息：

- 什么已经决定了：约束、承诺、先前设计、已有目录约定。
- 有什么证据：DDL、历史 SQL、报表、字段说明、业务口径、真实 Query、样例数据。
- 对领域已经了解什么：业务流程、对象关系、关键指标、常见决策。
- 还不知道什么：时间口径、有效状态、归因规则、主外键、权限边界等。
- 是否有之前尝试或相关探索可以借鉴。

使用差距图区分三类信息：

| 已知 | 假设 | 未知 |
| --- | --- | --- |
| 有证据支撑的事实 | 未验证的信念 | 需要调查才能确认的内容 |

不要混淆事实、假设和未知。大多数语义层错误来自把假设当事实。

### 步骤 2：质疑前提

在生成实体和概念之前，审查前提：

- 这是真的吗，还是只是从字段名推测？
- 什么情况会让这个前提为假？
- 如果这个前提错了，哪些 SQL、概念和回答会一起崩塌？
- 这个前提是业务共识，还是某个开发者的局部理解？

常见失败模式：

- “字段名叫 amount，所以它就是 GMV”。
- “status = paid 就代表有效订单”。
- “大家都这么问，所以这个概念一定存在”。
- “先建完整平台，再慢慢补口径”。
- “业务说挺好，所以定义没问题”。

输出前提时使用：

```text
前提：
1. [陈述] — 由 [证据 / 假设] 支持
2. [陈述] — 由 [证据 / 假设] 支持
```

如果关键前提只是未验证假设，先标记为 `assumptions`，必要时追问。

### 步骤 3：找到真正的问题

用户提出的需求可能不是本质问题。先重新框定：

- 用户真正想做的业务判断是什么？
- 如果不能直接做这个概念，有没有更小的可验证版本？
- 这个概念是业务确实需要，还是为了已有表结构找解释？
- 如果这是症状，根因是什么？
- 问“然后呢？”直到碰到真正重要的业务决策。

例如，“做销售语义层”不是问题；“让区域运营能用自然语言判断最近一周销售是否异常，并追溯到订单数、GMV、复购变化”才是更具体的问题。

### 步骤 4：生成备选方案

对非平凡设计，至少生成 2-3 个方案。一个方案不是选择，只是默认。

必须考虑：

1. 最小可行方案：最快验证、最少实体、最低承诺。
2. 理想架构方案：目录、schema、校验、运行时产物完整。
3. 创意 / 横向方案：不同切入方式，例如从历史 SQL、报表或典型 Query 反推概念。

决策矩阵：

| 方案 | 投入 | 风险 | 上限 | 适用情况 |
| --- | --- | --- | --- | --- |
| 最小可行 | 低 | 低 | 有限 | 信息不足、先验证价值 |
| 理想架构 | 高 | 中高 | 高 | 口径明确、准备长期维护 |
| 横向切入 | 中 | 中 | 未知 | 已有历史 SQL / 报表资产 |

### 步骤 5：用对抗性视角压测

攻击当前设计中最弱的地方：

- 什么会让这个语义层灾难性失败？
- 哪个业务角色会反对，为什么？
- 哪些 SQL 看起来对但业务含义错？
- 如果时间、数据质量、字段说明只有现在的十分之一，方案还能工作吗？
- 6 个月后回看，哪些地方会显得很蠢？

主动寻找反证，而不是只找支持当前设计的证据。分歧是信号：它揭示真正的不确定性在哪里。

### 步骤 6：决策分类

在做设计选择前，判断决策类型：

- 机械型：证据下一个选项明显正确，直接选。
- 品味型：合理人会分歧，明确权衡后下注。
- 可逆型：可以撤销或迭代，偏向行动。
- 不可逆型：会长期影响数据契约或生产回答，放慢并补证据。
- 承诺型：会形成团队约定或外部接口，要确认是否愿意被这个选择定义。

语义层中常见不可逆 / 承诺型选择：实体 ID 命名、指标口径、目录结构、运行时 JSON schema、对外 API 契约。

### 步骤 7：做出实际决策

使用以下格式记录重要设计：

```text
决策：[选择了什么]
理由：[为什么，引用证据和推理]
否决：[没有选什么以及为什么]
可逆性：[如果错了如何撤销]
下一步：[本周可以做的一件具体事情]
```

每次认真思考都必须产出一个具体后续行动。

### 步骤 8：事前验尸

在结束前想象方案已经失败：

- 具体是什么导致失败？
- 这个失败是否可预测？
- 忽略了哪些警告信号？
- 如果失败，下一步怎么修正？

事前验尸不是悲观，而是提前识别失败模式。

### 何时停止思考

满足以下条件即可停止：

- 已区分已知、假设、未知。
- 关键前提已确认或标记。
- 至少考虑过备选方案。
- 权衡和风险已经说清楚。
- 已知道下一步具体行动。

如果思考超过 2 小时仍没有决定，检查是否在用思考代替行动。

## 信息充分性门禁

评估以下输入：

- 业务场景和决策目标。
- 数据库 schema、字段类型、主键、外键和表关系。
- 字段业务含义。
- 指标口径：计算方式、过滤条件、时间语义、状态规则、归因规则、分类阈值。
- 典型用户问题：建议 5-10 个。
- SQL 方言和执行约束。
- 权限边界：可选；生产、多角色访问或敏感数据场景建议提供。

详细评分见 `references/information-checklist.md`。

如果只缺权限边界，不阻塞候选建模；应标记安全假设并建议后续补充。

## 建模规则

- 每个 L0 原子实体必须包含 SQL 或 SQL 模板。
- 每个 L0 原子实体必须包含来源表、粒度、参数、输出 schema 和事实解释模板。
- 概念层最多两层：`L1_business_concept` 和 `L2_decision_concept`。
- L1 业务概念只能依赖 L0 原子实体。
- L2 决策概念可以依赖 L1 概念，必要时可少量直接依赖 L0 原子实体。
- L2 决策概念不得依赖其他 L2 决策概念。
- 禁止循环依赖。
- 不要静默臆造业务定义；必须标记假设和确认问题。
- 优先创建业务稳定概念，而不是表结构的改名版本。
- 指标事实和解释规则要分离。
- 每个概念应包含同义词、业务关键词和典型用户问法。
- 每个重要实体和概念都应包含校验用例。

完整字段定义见 `references/entity-schema.md` 和 `references/concept-modeling-guide.md`。

## 输出格式与保存位置

使用 YAML 作为人工维护主格式，使用 JSON 作为运行时消费格式。

同时必须在项目根目录生成一份实体信息汇总 HTML：`semantic-layer-entities.html`。该 HTML 是面向开发者和业务评审的可读产物，不替代 YAML / JSON 源文件。

默认目录：

```text
./
├── semantic-layer-entities.html      # 根目录实体信息汇总 HTML
└── semantic-layer/{domain}/
    ├── README.md
    ├── semantic-layer.yaml
    ├── entities/
    │   ├── atomic.yaml
    │   └── concepts.yaml
    ├── sql/
    │   └── *.sql
    ├── mappings/
    │   └── query-examples.yaml
    ├── tests/
    │   └── semantic-layer-tests.yaml
    └── dist/
        └── semantic-layer.json
```

如果项目已有 `config/`、`sematic-layer/`、`metadata/` 或 `knowledge/` 等目录约定，优先遵循项目约定。

### 根目录实体信息 HTML 要求

最终交付时，必须在项目根目录生成 `semantic-layer-entities.html`，将 entities 相关信息整理成一份可浏览的 HTML。HTML 至少包含：

- 页面标题、业务域、生成时间、准备度评分和待确认项摘要。
- L0 原子实体总览表：`id`、名称、业务含义、来源表、粒度、SQL 模板引用、SQL 模板原文、参数、输出字段、假设。
- L1 业务概念总览表：`id`、名称、依赖的 L0 原子实体、组合规则、关键词、示例问法、回答模板。
- L2 决策概念总览表：`id`、名称、依赖的 L1 概念 / L0 原子实体、组合规则、关键词、示例问法、回答模板。
- 概念依赖图或树形结构：展示 `L2 → L1 → L0 原子实体` 的追溯关系。
- SQL 模板原文：每个 L0 原子实体都必须展示完整 SQL 模板原文；如果实体使用 `sql_template_ref` 指向 `.sql` 文件，生成 HTML 时必须读取该 SQL 文件全文并嵌入页面，而不能只展示文件路径或 SQL 摘要。
- 风险与假设：展示 `assumptions`、缺失口径、draft SQL、权限边界状态。

HTML 应使用内联 CSS，不能依赖外部资源；内容应来自 `semantic-layer/{domain}/entities/atomic.yaml` 与 `semantic-layer/{domain}/entities/concepts.yaml`，不得只写概念性说明。

完整文件契约见 `references/output-layout.md`。

## 校验要求

交付前至少做结构校验：

- `depends_on` 中的所有引用都存在。
- L1 只依赖 L0 原子实体。
- L2 不依赖 L2。
- 不存在循环依赖。
- L0 原子实体包含 SQL 模板和参数声明。
- Query 映射引用的概念和实体都存在。
- 项目根目录存在 `semantic-layer-entities.html`，且其中包含 L0 原子实体、L1 概念、L2 概念、依赖关系摘要和每个 L0 原子实体的 SQL 模板原文。
- 如果提供了权限边界，敏感字段必须被禁用、脱敏或明确标记。

本地存在语义层文件时，使用 `scripts/validate_semantic_layer.py` 校验。

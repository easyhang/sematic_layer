# 信息充分性检查表

在生成正式语义层定义之前，先使用本检查表判断输入是否足够。

## 评分规则

总分 100 分：

| 维度 | 分值 | 说明 |
| --- | ---: | --- |
| 业务目标清晰度 | 20 | 是否说明业务域、使用者、决策目标和成功标准。 |
| 数据表结构完整度 | 20 | 是否提供表、字段、类型、主键、样例数据等。 |
| 字段业务含义完整度 | 15 | 关键字段是否有业务解释。 |
| 表关系明确度 | 15 | 是否说明 join key、基数关系、粒度和重复风险。 |
| 指标口径明确度 | 15 | 是否说明计算、过滤、时间、状态、归因、分类阈值。 |
| 典型 Query 覆盖度 | 10 | 是否提供 5-10 个真实或接近真实的用户问题。 |
| 权限边界说明 | 5 | 可选安全加分项；缺失时不阻塞候选建模。 |

## 准备度等级

- 85-100：可以构建正式初版语义层。
- 70-84：可以构建草案，但必须标记假设和待确认项。
- 50-69：只能生成候选实体，不建议直接用于生产。
- 0-49：信息不足，应先追问再建模。

## 候选建模的最小输入

- 业务场景。
- 相关数据表结构或 DDL。
- 核心指标口径，或至少给出需要确认的指标名称。
- 时间字段和默认时间语义。
- 表关系；如果未知，必须明确标记未知。
- 至少 3 个典型用户问题。

## 权限边界是可选项

权限边界不是草案语义层的必需输入，但在以下场景建议提供：

- 语义层将用于生产环境。
- 不同角色存在不同数据访问范围。
- 可能涉及敏感字段：手机号、邮箱、身份证号、详细地址、薪资、成本、毛利、医疗数据、金融账户等。
- 智能体可能回答明细级问题或导出数据。

如果缺失，写入：`permission_boundary_status: not_provided`，并列出潜在安全风险。

## 追问优先级

优先追问影响 SQL 正确性的内容：

1. 哪些表和字段是权威来源？
2. 每个指标应使用哪个时间字段？
3. 哪些状态算有效，哪些应排除？
4. 表之间如何关联，粒度是什么？
5. 核心指标如何计算？
6. 是否能提供 5-10 个代表性用户问题？
7. 可选：如果涉及生产、多角色或敏感数据，是否有角色、行级、字段级、明细级和禁止查询规则？

每轮最多追问 5-8 个关键问题，避免一次性问题过多。

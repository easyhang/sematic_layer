# 概念建模指南

语义层只有三个建模层级：

1. L0 原子实体：可直接落到 SQL 的数据事实。
2. L1 业务概念：由 L0 原子实体组合出的业务观察。
3. L2 决策概念：由 L1 概念组合出的业务判断、分类、预警或决策。

禁止创建 L3 概念。

## 层级判断规则

使用以下规则决定对象应该放在哪一层：

- 如果它是表字段、直接聚合或单一稳定事实：建为 L0 原子实体。
- 如果它组合多个 L0 原子实体，用于解释一个业务观察：建为 L1 概念。
- 如果它组合多个 L1 概念，用于判断、分类、排序、预警或决策：建为 L2 概念。
- 如果看起来需要 L3，把它作为最终回答编排逻辑，或拆成多个 L2 概念。

## 好的 L1 概念

- 销售趋势：每日 GMV + 每日支付订单数。
- 复购表现：客户订单数 + 客户复购率。
- 渠道转化表现：曝光、访问、加购、支付订单等事实。
- 商品销售表现：商品销量 + GMV + 退款金额。

## 好的 L2 概念

- 销售健康度：销售趋势 + 客单价表现 + 复购表现。
- 高价值客户：客户价值表现 + 复购表现 + 活跃稳定性。
- 流失风险客户：历史价值 + 近期不活跃 + 复购下降。
- 异常商品预警：销售下滑 + 退款上升 + 库存风险。

## 反模式

- 把表名改个名字就当概念，例如 `orders_concept`。
- 创建无法追溯到 SQL 的概念。
- 一个概念混合多个无关业务含义。
- 让 L2 依赖 L2。
- 创建没有阈值、样例或回答模板的模糊概念。
- 把口径假设藏在自然语言里，而不是写入 `assumptions`。

## 常见组合模式

- `metric_group`：把多个事实组合成一个观察。
- `trend`：比较当前周期和上一周期。
- `classification`：按阈值打标签。
- `health_score`：综合多个 L1 概念形成健康度判断。
- `segment`：按规则分群。
- `exception_detection`：识别异常变化。
- `funnel`：建模分阶段转化。

## 必需可追溯信息

每个概念都应包含：

- `depends_on`
- `composition_rule`
- `business_keywords`
- `example_queries`
- `answer_template` 或 `interpretation_rule`
- `assumptions`
- `validation_cases`
